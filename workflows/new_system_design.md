# 路径1：新系统设计

## 认知框架说明

**本工作流基于**：四层嵌套核心循环 + System Stack方法

**核心假设**：
- 系统有明确的目标和体验需求，可以通过结构化设计实现
- 系统有可识别的核心动作，可以通过四层嵌套模型组织
- 系统可以拆解为子系统，通过单向数据流组织

**适用场景**：
- 设计新系统，有明确的设计目标和约束
- 系统核心是层级结构（核心层→系统层→子系统层）
- 需要产出完整的GDD文档

**失效信号**（出现以下情况时应重新审视框架）：
- 系统核心是涌现式玩法而非层级结构（如沙盒游戏的UGC系统）
- 无法识别明确的核心动作（如叙事驱动游戏）
- 系统间依赖关系复杂，无法形成单向数据流
- 连续2个步骤填不满完成标准

**替代框架**：
- 如果是涌现式玩法，考虑"涌现式系统分析"框架
- 如果是叙事驱动，考虑"叙事-玩法整合"框架

## 触发条件
用户描述需要设计一个新系统（如"设计一个背包系统""设计一个技能树"）

## 工作流步骤

### 步骤0：初始化产出文档

**执行模式**：A-自主

**操作说明**：
- 参照 `../references/template_gdd.md` 模板，创建文档文件
- 命名为 `[系统名称]_设计文档.md`
- 模板包含9个部分：Objectives → Overview → Mechanics → Data Flow → UI → Edge Cases → Configuration → Metrics & Analytics → Technical Notes
- 后续每步完成后按需更新此文档

**完成标准**：
- 文档已创建
- 模板结构完整（9个部分都有占位）

**验证问题**：
- 文档是否已创建并可编辑？
- 模板结构是否完整？

---

### 步骤1：需求拆解

**执行模式**：H-人类主导

**操作说明**：
- 提供引导问题，等待人类回答
- 整理人类输入，填充到产出文档的"Objectives"部分

**引导问题**：
1. 这个系统要解决什么问题？（不是"让玩家觉得好玩"，而是具体的问题）
2. 玩家在使用这个系统时应该感受到什么？（具体的情感体验）
3. 如何判断这个系统设计成功？（可度量的标准）
4. 有什么约束条件？（平台、技术、美术风格、目标受众）
5. 这个系统包含什么、不包含什么？（明确的边界）

**完成标准**（必填字段）：
- 核心问题：[具体描述]
- 体验目标：[具体描述]
- 成功标准：[可度量描述]
- 约束条件：[平台/技术/美术/受众]
- 系统边界：[包含/不包含列表]

**验证问题**：
- 系统目标是否可度量？（不是"让玩家觉得好玩"）
- 约束条件是否具体？（不是"性能要好"，而是"移动端60FPS"）
- 系统边界是否明确？（不是"大概包括这些"）

**框架质疑触发**：
- 如果人类无法回答"核心问题"，可能任务定义不清晰
- 如果人类说"不知道要什么体验"，可能需要先做竞品研究

---

### 步骤2：核心循环构建

**执行模式**：C-协作

**操作说明**：
- AI基于步骤1的需求，出初版四层嵌套核心循环
- 明确标注需要人类补充的部分（特别是体感判断）
- 等待人类补充后迭代

**AI初版产出**：
```
## 核心循环设计（初版）

### 秒级操作（Core Action）
- 核心动作：[AI推测]
- 操作频率：[AI推测]
- 验证标准：[需要人类体验判断]

### 分钟级活动（Activity）
- 活动定义：[AI推测]
- 活动时长：[AI推测]
- 活动结构：[AI推测]
- 验证标准：[需要人类体验判断]

### 会话级目标（Session Goal）
- 目标定义：[AI推测]
- 目标类型：[AI推测]
- 验证标准：[需要人类体验判断]

### 元循环（Meta Loop）
- 循环定义：[AI推测]
- 进度类型：[AI推测]
- 验证标准：[需要人类体验判断]
```

**需要人类补充的部分**：
- 秒级操作是否"空房间也好玩"？（需要体感判断）
- 分钟级活动的节奏是否舒适？（需要体感判断）
- 会话级目标是否有成就感？（需要体感判断）
- 元循环是否有长期动力？（需要体感判断）

**完成标准**（必填字段）：
- 四层嵌套完整（每层都有）
- 每层有明确的输入/输出状态
- 每层有可测试的验证标准
- 层与层之间有清晰的因果关系

**验证问题**：
- 四层嵌套是否完整？
- 每层的验证标准是否可测试？（不是"应该好玩"）
- 层与层之间是否有因果关系？

**框架质疑触发**：
- 如果无法识别核心动作，可能系统不适合用四层嵌套模型
- 如果人类说"这个循环感觉不对"，可能需要重新审视需求

---

### 步骤3：System Stack子系统拆解

**执行模式**：C-协作

**操作说明**：
- AI基于步骤2的核心循环，用System Stack方法拆解子系统
- 产出初版系统架构
- 等待人类审阅后迭代

**AI产出**：
1. 动词提取：从核心循环中提取所有玩家动作
2. 动词→系统映射：对每个动词追问"需要什么系统支持？"
3. 依赖分组：按依赖关系分层，确保单向数据流

**完成标准**（必填字段）：
- 所有动词都有对应的系统支撑
- 依赖关系是单向的（没有循环依赖）
- 每个系统的职责明确（一句话能说清楚）
- 系统清单完整（名称、类型、依赖、职责）

**验证问题**：
- 是否有遗漏的动词？
- 是否有循环依赖？
- 每个系统的职责是否清晰？

**框架质疑触发**：
- 如果系统间依赖关系复杂，无法形成单向数据流，可能需要换架构方法
- 如果动词数量过多（>20个），可能系统过于复杂，需要拆分

---

### 步骤4：GDD撰写

**执行模式**：A-自主

**操作说明**：
- 基于前3步的产出，按GDD模板撰写完整文档
- 每个系统内部结构：Objectives → Overview → Mechanics → Data Flow → UI → Edge Cases → Configuration
- 必须包含配置表结构

**完成标准**（必填字段）：
- 每个系统内部结构完整（7个部分都有）
- 机制描述精确（有具体的规则和公式）
- 数据流关系明确（没有模糊的"可能来自"）
- 配置表结构完整（字段名、类型、说明、示例值）

**验证问题**：
- 7个部分是否都有？
- 机制是否有具体公式？
- 数据流是否明确？
- 配置表是否完整？

---

### 步骤5：逻辑验证

**执行模式**：A-自主

**操作说明**：
- 检查系统设计的逻辑自洽性
- 检查与其他系统的耦合度
- 检查边界条件是否覆盖完整

**完成标准**（必填字段）：
- 逻辑自洽性检查通过
- 耦合度评估为"低"或"中"
- 边界条件覆盖度100%

**验证问题**：
- 所有检查项是否都通过？
- 耦合度是否可接受？
- 边界条件是否覆盖完整？

**框架质疑触发**：
- 如果耦合度为"高"，可能需要重新设计系统架构
- 如果边界条件无法覆盖，可能系统设计有缺陷
